# Список задач — кто что делает прямо сейчас

Смысл файла: чтобы два разных чата с Claude случайно не работали
над одним и тем же местом в проекте одновременно и не затирали
правки друг друга.

## Как этим пользоваться (простыми словами)

1. Начинаешь новый чат с Claude по новой задаче (баг, новый материал,
   правка чертежа) — сначала загляни в этот файл. Если похожая задача
   уже стоит в списке "в работе" — доделай её сначала, не начинай вторую
   параллельно в той же области.
2. Когда чат начинает задачу — просишь Claude добавить строку сюда
   (раздел "В работе"), с названием "подсобки" (ветки) и коротким
   описанием.
3. Когда задача доделана и перенесена в основной этаж (в main) —
   Claude переносит строку в раздел "Готово" и убирает.
4. Если задачу забросили на середине — тоже отметь, чтобы не думать
   потом "а это что за недоделка".
5. **Перед любым пушем — обязательно `npm test && npm run build`.**
   Проект уже большой (300+ тестов), руками все связи не удержать —
   тесты это делают за секунды. Если хоть один тест красный или
   сборка падает — пуш не делаем, разбираемся.
6. **После мержа своей ветки в `main` — сначала `git fetch origin &&
   git reset --hard origin/main`, и ТОЛЬКО ПОТОМ мержить свою ветку.**
   Причина: пока один чат работает, в main может прилететь что-то
   от ДРУГОГО параллельного чата (bugfix, новая фича) — это уже
   случалось на практике. Мерж "на глаз" без свежего fetch рискует
   тихо потерять чужие изменения или объединить несовместимые правки
   без предупреждения.
7. **После мержа (когда обе ветки объединены) — прогнать
   `npm test && npm run build` ЕЩЁ РАЗ, уже на объединённом коде.**
   Правило 5 проверяет твою ветку саму по себе, это отдельное —
   что твоя ветка и то, что притащил другой чат, не мешают друг
   другу вместе. "У меня было чисто" не гарантирует "у нас вместе
   чисто" — проверять надо именно после объединения.
8. **После пуша в `main` — проверить, что деплой на GitHub Pages
   реально прошёл, а не просто "запушил и забыл".** `npm test`
   и `npm run build` проверяют только код — деплой может упасть
   отдельно, уже на стороне GitHub (бывало на практике: "Deployment
   failed, try again later" — обычно временный сбой самого GitHub
   Pages, не связан с кодом). Проверить статус последнего workflow
   run в репозитории (вкладка Actions, или через API:
   `curl -H "Authorization: Bearer <токен>" -H "User-Agent: check"
   "https://api.github.com/repos/SergeyZuda1982/wall-calc/actions/runs?per_page=1"`
   — смотреть на `"conclusion":"success"`). Если упал — не значит,
   что код плохой: обычно достаточно перезапустить деплой (пустой
   коммит `git commit --allow-empty` + push, или кнопка "Re-run" в
   Actions на GitHub). Считать задачу закрытой только после
   успешного деплоя, не после успешного пуша.

## Правило веток (для Claude, техническое)

Каждая новая задача — отдельная ветка от актуального main:

```bash
git fetch origin && git checkout main && git reset --hard origin/main
git checkout -b <тип>/<короткое-название>
# тип: fix/ — багфикс, feat/ — новая фича, docs/ — только конспект/документация
```

Мерж в main — только после `npm test && npm run build` без ошибок,
и повторно после самого мержа (на случай тихого конфликта).

# Список задач — кто что делает прямо сейчас

Смысл файла: чтобы два разных чата с Claude случайно не работали
над одним и тем же местом в проекте одновременно и не затирали
правки друг друга.

## Как этим пользоваться (простыми словами)

1. Начинаешь новый чат с Claude по новой задаче (баг, новый материал,
   правка чертежа) — сначала загляни в этот файл. Если похожая задача
   уже стоит в списке "в работе" — доделай её сначала, не начинай вторую
   параллельно в той же области.
2. Когда чат начинает задачу — просишь Claude добавить строку сюда
   (раздел "В работе"), с названием "подсобки" (ветки) и коротким
   описанием.
3. Когда задача доделана и перенесена в основной этаж (в main) —
   Claude переносит строку в раздел "Готово" и убирает.
4. Если задачу забросили на середине — тоже отметь, чтобы не думать
   потом "а это что за недоделка".
5. **Перед любым пушем — обязательно `npm test && npm run build`.**
   Проект уже большой (300+ тестов), руками все связи не удержать —
   тесты это делают за секунды. Если хоть один тест красный или
   сборка падает — пуш не делаем, разбираемся.
6. **После мержа своей ветки в `main` — сначала `git fetch origin &&
   git reset --hard origin/main`, и ТОЛЬКО ПОТОМ мержить свою ветку.**
   Причина: пока один чат работает, в main может прилететь что-то
   от ДРУГОГО параллельного чата (bugfix, новая фича) — это уже
   случалось на практике. Мерж "на глаз" без свежего fetch рискует
   тихо потерять чужие изменения или объединить несовместимые правки
   без предупреждения.
7. **После мержа (когда обе ветки объединены) — прогнать
   `npm test && npm run build` ЕЩЁ РАЗ, уже на объединённом коде.**
   Правило 5 проверяет твою ветку саму по себе, это отдельное —
   что твоя ветка и то, что притащил другой чат, не мешают друг
   другу вместе. "У меня было чисто" не гарантирует "у нас вместе
   чисто" — проверять надо именно после объединения.
8. **После пуша в `main` — проверить, что деплой на GitHub Pages
   реально прошёл, а не просто "запушил и забыл".** `npm test`
   и `npm run build` проверяют только код — деплой может упасть
   отдельно, уже на стороне GitHub. Проверить статус последнего
   workflow run в репозитории (вкладка Actions, или через API:
   `curl -H "Authorization: Bearer <токен>" -H "User-Agent: check"
   "https://api.github.com/repos/SergeyZuda1982/wall-calc/actions/runs?per_page=1"`
   — смотреть на `"conclusion":"success"`). Считать задачу закрытой
   только после успешного деплоя, не после успешного пуша.

   **05.07.2026 — найдена и устранена настоящая причина учащения этих
   сбоев:** дело не в случайном сбое GitHub, а в том, что несколько
   параллельных чатов пушат в `main` почти одновременно — их деплои
   на Pages конфликтуют друг с другом за право публикации, и один из
   них падает. В `.github/workflows/deploy.yml` добавлен блок
   `concurrency: {group: "pages", cancel-in-progress: false}`
   (официальная рекомендация самого GitHub) — теперь такие деплои
   встают в очередь вместо конфликта. Если падения продолжатся ПОСЛЕ
   этого — причина уже другая, не эта гонка, копать заново.
   ⚠️ Важно: PAT для пуша в `main` теперь должен включать право
   **workflow** (обычного `repo` недостаточно) — GitHub отдельно
   защищает файлы `.github/workflows/*` от токенов без этого scope.

## Раскрой ГКЛ поворачивался ради экономии материала (16.07.2026)

## 15.07.2026 — фикс: раскрой ГКЛ по контуру вылезал за пределы на частичных вогнутых выемках

Пользователь прислал 3 скриншота (2D-схема + 3D) сложного контура потолка
П113 (пятиугольная пристройка слева + отдельные выступы, много углов):
раскрой листов ГКЛ в 3D выглядел как разлетевшиеся в стороны обломки, не
совпадающие ни с контуром, ни с сеткой каркаса профиля (та сама рисовалась
верно).

Причина — в `calcPolygonSheetLayout.ts` (calcLayerDetailed): ширина
покрытия полосы (band, толщиной до 1200мм) определялась ОДНОЙ скан-линией
через её центр. Это верно только там, где ширина не меняется вдоль длины
полосы (документированное "осознанное упрощение" в шапке файла). Если
вогнутая выемка срезает полосу ЧАСТИЧНО по толщине (не всю 1200мм, а
кусок) и центральная скан-линия проходит МИМО выемки — весь кусок листа на
этом участке длины получал полную (неверную) ширину и физически вылезал за
контур. На синтетическом тесте (узкий вогнутый контур 1200×6000 с выемкой
до 700мм) кусок вылезал на 500×1500мм за пределы контура.

Фикс — тот же приём, что уже используется для несущего профиля П113
(`splitSegmentAtCuts` в `calcPolygonP113Frame.ts`): между соседними
вершинами контура ширина покрытия полосы не может измениться (граница
локально прямолинейна). Добавлены `bandCriticalXs`/`splitAtCriticalXs`/
`widthRangesAtX` — каждый кусок листа дополнительно режется по критическим
длинам (вершины контура внутри полосы), и для каждого под-куска ширина
покрытия пересчитывается по его СОБСТВЕННОЙ скан-линии, а не унаследована
от центра всей полосы. Для прямоугольных контуров (без вершин внутри
полосы) поведение не меняется — критических точек нет, старые тесты не
затронуты.

Остаточная погрешность на диагональных (не осе-параллельных) участках
контура (например, острый угол пятиугольной пристройки) сохраняется —
документированное упрощение, теперь единицы % площади куска у самого угла,
а не полный кусок листа целиком мимо контура.

Проверено на реконструкции контура пользователя (по скриншотам, +отдельным
скриптом с shapely, вне репозитория): сумма площадей кусков раскроя точно
совпадает с площадью контура, ни один кусок не выходит за границы.

2 новых теста (сумма площадей кусков = площадь контура по формуле
шнурования; ни один кусок не шире фактической ширины контура на своём
участке длины).

Ветка `fix/ceiling-polygon-sheet-width-notch`, PR #20, смержен в main
(`cacce02`). 1023/1023 тестов, build чист.

## 15.07.2026 — фича: независимый шаг подвесов (a) от шага несущего профиля (b) в user-режиме + шаг основного профиля 400мм

Пользователь: у П112/П113 в реальной практике объекта шаг подвесов и шаг
несущего профиля — РАЗНЫЕ, независимо выбираемые монтажником величины,
хотя каркас делаем "по КНАУФ" — по факту монтажники чаще собирают иначе.
Пример с объекта: потолок П113, шаг основного профиля c=400мм — при этом
ГКЛ (монтаж "вдоль") ложится прямо на основной профиль на 0-400-800-1200мм
от стены (0 — периметральный ПН 28×27), несущий профиль в этом случае —
только короткие вставки у торцевых стыков ГКЛ-листов, с произвольным
шагом. Шаг подвесов (типичная практика — 800мм) — тоже произвольная,
независимая величина.

До этой правки `resolveFrameParams()` в `layoutMode='user'` молча
приравнивал stepA (шаг подвесов) к stepB (шаг несущего профиля) —
пользователь физически не мог задать их по отдельности, в UI было только
одно поле «Шаг несущего». Геометрия каркаса (calcP112FrameGeometry/
calcP113FrameGeometry/calcPolygonP112Frame/calcPolygonP113Frame) уже давно
умела принимать stepA отдельно (`extra.stepA`) — проблема была только в
слое resolveFrameParams + UI + типах, сама расчётная геометрия не
потребовала изменений.

Сделано:
- `CeilingStep`: добавлено 400мм (`CEILING_STEP_OPTIONS`).
- `CeilingSpec.stepA?: number` — новое поле, шаг подвесов независимо от
  `stepB`, только для `layoutMode='user'`. Дефолт (если не задан) —
  `DEFAULT_USER_HANGER_STEP_MM = 800` (подтверждено пользователем как
  типичная практика для П112 и П113). `layoutMode='knauf'` не затронут —
  там шаг подвесов всегда по официальной таблице.
- `resolveFrameParams()`: новый параметр `userStepA`; в user-режиме
  `stepA = userStepA ?? 800`, больше не привязан к `stepB`.
- Проброшено во все 8 точек вызова `resolveFrameParams` (CeilingCalc.tsx×2,
  CeilingEntityMesh.tsx, Scene3D.tsx, calcCeiling.ts×4) и во все 4 места
  сборки `CeilingSpec` из формы (сохранение в Room/Ceiling, превью).
- UI (`CeilingCalc.tsx`, режим "Пользовательский"): отдельное поле «Шаг
  подвесов (a)» рядом с «Шаг несущего (b)», плейсхолдер-дефолт 800.

⚠️ Открытый вопрос на будущее (пользователь описал, но не просил
реализовывать в этой правке): в описанном сценарии (c=400, ГКЛ "вдоль")
несущий профиль физически ставится ТОЛЬКО у торцевых стыков ГКЛ-листов —
то есть его позиции зависят от раскроя листов (calcPolygonSheetLayout), а
не от равномерного шага b. Сейчас несущий профиль всё ещё считается как
равномерная сетка с шагом b (просто теперь свободно выбираемым числом) —

2 существующих теста в calcP112Frame.test.ts переписаны под новое
поведение (раньше проверяли stepA=stepB), добавлено 3 новых теста + 1 на
CEILING_STEP_OPTIONS.

Ветка `feat/ceiling-independent-hanger-step`, PR #24, смержен в main
(`17ca55c`). 1059/1059 тестов, build чист.

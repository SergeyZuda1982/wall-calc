## Стратегическое видение проекта (озвучено в этом чате, важно держать в голове)

Пользователь строит wall-calc не просто как калькулятор ГКЛ, а как систему
управления объектом для всей команды:
- **Руководитель** — прогресс по всем объектам, % выполнения работ
- **Прораб** — где затык, где можно продолжать, где выполнено
- **Снабжение** — сколько материала ещё нужно завезти (в перспективе — учёт
  закупленного/остатков относительно выработки)
- **Монтажник** — оптимальная раскладка профиля, суммы по участкам работ

Базовая инвариантная модель объекта: заходим на объект (новостройка или
демонтаж/частичный демонтаж) — **не меняется периметр (наружные стены),
уровень пола, уровень потолка, колонны**. Всё остальное (перегородки,
облицовки, полы, потолки) — то, что демонтируется/строится/переделывается.
Всегда есть предположение "что-то к чему-то примыкает" — щелей между
конструкциями быть не должно, и это касается ВСЕХ типов стыков
(перегородка-перегородка, стена-стена, вокруг колонны).

**Роадмап, согласованный в этом чате** (архитектурные слои, не строго
последовательные, но с найденной взаимосвязью):
1. **Слой 1** — модель данных: капитал/изменяемая + статус работ — ГОТОВО
2. **Слой 2** — универсальный снап с приоритетом капитал/изменяемая — ГОТОВО
3. **Слой 3** — типовые колонны (библиотека, вставка кликом) — НЕ НАЧАТО
   (пока колонна = замкнутый контур 4 capital-линий с чекбоксом isColumn — работает,
   но не библиотека типоразмеров)
4. **Переводчик `PlanLine → SurfaceSheetInput`** — чтобы План считал не только
   площадь, а реальный раскрой листов/саморезы через уже существующий
   `calcProjectSheetLayout.ts` (движок абстрагирован от типа поверхности,
   писать новый не нужно) — НЕ НАЧАТО. Это одновременно закрывает: расчёт
   колонны одним проходом вместо 4 расчётов вручную, и главную цель ещё
   более ранних чатов "калька PDF → материалы → смета"
5. Раскладка плитки для пола/потолка (протяжка мышью от угла до угла) — НЕ
   НАЧАТО, но фундамент готов: `addRoom()` уже автоматически считает
   areaM2/perimeterMm при замыкании периметра
6. Роли/дашборды — упирается в Supabase (см. выше) + во все слои 1-4 снизу

## Что сделано в этом чате (хронологически, коммиты f70fe9d → 11705cf)

### 1. Раскладка вкладки «План» на всю ширину экрана
`App.tsx`: `maxWidth` обёртки условный — `900` для форм Перегородки/
Облицовка/Потолки, `none` для вкладки «План». Скрыт дублирующий блок
«Объект/Нет перегородок/Нет облицовок» на вкладке План (дублировал имя
объекта из шапки, отжирал вертикальное место). Высота канваса
`calc(100vh - 100px)`.

### 2. Колонна = замкнутый контур со штриховкой
`Room.isColumn?: boolean` в types. Клик по центру замкнутого контура (в
режиме "Двигать"/select) открывает мини-панель справа: название, чекбокс
"Это колонна (заштриховать)", площадь/периметр, удаление. При isColumn —
диагональная штриховка внутри контура (клипается по границе полигона),
периметр скрыт из подписи (не нужен для колонны).

**Важный фикс клика для маленьких контуров**: у маленькой колонны (типа
0.2 м²) хитзона стен (24px) перекрывает буквально всю площадь контура —
клик всегда попадал в ближайшую стену раньше, чем в саму комнату. Решение:
маленькая кнопка-маркер (⛶/▦, белый кружок ~9px) рисуется В ЦЕНТРЕ контура
ПОВЕРХ всех линий (z-order — рендерится после блока "Линии"), гарантированно
кликабельна независимо от размера контура. Полигон комнаты сам по себе
больше НЕ кликабелен (listening=false) — весь клик идёт через маркер.

### 3. Кнопка "Показать всё" + автоцентрирование подложки
Пользователь терял из виду PDF-подложку после панорамирования (не мог
найти, куда она делась). `fitToContent()` — вычисляет bbox всех линий +
подложки, вписывает в canvasW×CANVAS_H с падингом 60px, центрирует.
Кнопка "🎯 Показать всё" в шапке плана. Плюс автовызов через 50мс после
загрузки/смены подложки (useEffect на `floorPlan?.backgroundImage?.dataUrl`).

### 4. Слой 1 — категория и статус работ на линиях плана
`types/index.ts`: `LineCategory = 'capital' | 'mutable'`,
`WorkStatus = 'demolition' | 'existing' | 'planned' | 'in_progress' | 'done'`.
`PlanLine.category?`, `PlanLine.workStatus?`.

Дефолты: `defaultCategory(type)` — `wall_existing` → capital, всё
остальное → mutable. `defaultStatus(category)` — capital → existing,
mutable → planned. Проставляются во всех местах создания линии (обычный
клик, замыкание цепочки, параллельная копия).

UI в инспекторе линии: кнопки-тумблер 🔒Капитал/✏️Изменяемая + выпадающий
список статуса (Под снос/Существует/Запланировано/В работе/Выполнено).
На канвасе у mutable-линий — маленькая цветная точка-индикатор статуса
рядом с подписью (капитал не показывает — всегда "существует", не несёт
пользы).

### 5. КРУПНЫЙ БАГФИКС: T-стык не распознавался для толстых стен
`wallJoin.ts` до этого чата **не имел ни одного теста** (самый хрупкий
модуль в проекте). Написан `wallJoin.test.ts`.

Найден реальный баг: детекция T-стыка сравнивала расстояние от конца
примыкающей линии до **ОСИ** главной стены (допуск 3px), а `snapPoint()`
уже сессии назад был переделан ставить точку на **ГРАНЬ** (см. baг №2
в старом конспекте — исправлен раньше). Для любой стены толще ~30мм
(т.е. почти всех реальных ГКЛ-конструкций) T-стык физически не находился —
торец рисовался там, где не должен, полигон заливки не сглаживался.

**Фикс**: допуск теперь — вся полоса толщины стены
(`distToAxis <= b.halfPx + JOIN_EPS`), а не только 3px от оси. Это же
даёт самокоррекцию "случайно завёл линию на монолит" — даже если конец
попал не точно на грань, а куда-то в середину тела стены, applyT всё
равно корректно подтягивает его к правильной грани (проверено тестом).

### 6. Слой 2 — приоритет капитал/изменяемая в T-стыках
`WallForJoin.category?: 'capital' | 'mutable'` в `wallJoin.ts`.
`attachedYieldsToMain(attached, main)` — возвращает `false`, если
attached.category === 'capital' И main.category === 'mutable' (капитал
никогда не становится "обрезаемой" стороной, даже если его конец
геометрически лёг на тело mutable-стены — редкий, но возможный случай
ошибочного черчения). Между двумя capital или двумя mutable — приоритета
нет, обычная симметричная геометрия. Без category (старые данные) —
работает как раньше. 5 тестов на все комбинации.

`FloorPlan.tsx`: `category: l.category ?? defaultCategory(l.type)`
прокидывается в массив `walls[]` перед `computeWallJoins()`.

**Важно**: L-угол (`applyL`, конец=конец) category НЕ учитывает — это
осознанно, там нет коллизии "кто главный", оба физически существуют и
просто мЯтруются в реальный угол. Приоритет нужен только в T (примыкание
к телу другой стены).

### 7. Линия "Тип неизвестен" — толщина и превью
`wall_existing:unknown` в `constructionTaxonomy.ts`: `strokeWidth 4→1.5`
(было в 2.5 раза толще concrete/block, из-за пунктира+круглых колпачков
выглядело как "цепочка капсул", а не тонкая линия).

Заодно найден сопутствующий баг: превью рисования (пока линия ещё не
отпущена) вообще игнорировало материал — всегда фиксированная толщина
5px и пунктир [6,4], независимо от того что задано в справочнике.
Исправлено: превью использует `previewVis.strokeWidth`/`previewVis.dash`
(с той же конвертацией `dash ?? undefined`, что и в финальном рендере —
`null` означает СПЛОШНУЮ линию, важно не перепутать с "используй дефолт").

## Известные особенности / договорённости для будущих сессий

1. При наличии подложки PDF ограничение "сначала периметр" снято —
   можно рисовать любую конструкцию в любом порядке (для сценария кальки).

2. T-снап целится в грань стены, не в ось. Длина "в свету" автоматически.
   wallJoin теперь тоже корректно ловит T-стык по всей полосе толщины
   стены (не только по оси) — коммит 12e6634.

3. Offset проёма отсчитывается от ПЕРВОЙ точки линии (x1,y1).

4. drawHeightMm — глобальный, не привязан к материалу.

5. Подложка PDF рендерится один раз при загрузке, не двигается независимо
   от плана, но панорамируется вместе с планом. Если потеряна из виду —
   кнопка "Показать всё" в шапке плана.

6. Wall-join расширяет ось линии под стыки — влияет на
   `computeOpeningSegments`, которая берёт исходные (не расширенные)
   `l.x1/l.y1` для offsetMm проёма, но расширенные `fax1..fay2` для
   крайних сегментов заливки. Трогаете один модуль — проверяйте другой.

7. `maxWidth: 900` у обёртки контента остаётся для форм
   Перегородки/Облицовка/Потолки — сознательно не тронут. Убран только
   для вкладки «План».

8. **Категория линии (`category`) не путать с типом (`type`)**: тип —
   это ЧТО это физически (перегородка/облицовка/потолок/пол/существующая
   конструкция), категория — МОЖНО ЛИ это трогать (капитал/изменяемая).
   Дефолт капитала завязан на тип `wall_existing`, но пользователь может
   переключить вручную в инспекторе (кнопки 🔒/✏️) — например если
   периметр частично сносится, или наоборот что-то остаётся как каптал.

9. **Колонна = Room с `isColumn: true`**, не отдельная сущность. Библиотека
   типовых колонн (слой 3) пока не начата — сейчас каждую колонну нужно
   обвести вручную 4 линиями и отметить чекбоксом.

10. **wallJoin.ts теперь покрыт тестами** (9 штук, `wallJoin.test.ts`) —
    любые дальнейшие правки геометрии стыков можно и нужно проверять
    тестами перед пушем, это исторически самый баговый модуль проекта.

11. Workflow макета UI-раскладки от руки на бумаге подтверждён рабочим —
    пользователь фотографирует эскиз, Claude читает и переводит в
    CSS/flex прямо в чате. НЕ путать с другой (нереализованной) идеей —
    автораспознавание ПЛАНА ПОМЕЩЕНИЯ из фото внутри самого приложения
    (это отдельная, кратно более сложная задача, Vision API как часть
    продукта, не начата и не обсуждалась детально).

## Архитектура данных (types/index.ts) — актуальный срез

```typescript
export type LineCategory = 'capital' | 'mutable'
export type WorkStatus = 'demolition' | 'existing' | 'planned' | 'in_progress' | 'done'

export interface PlanLineSpec {
  material: string
  subtype?: string
  boardSubtype?: GklSubtype
  layers?: 1 | 2
}

export interface PlanOpening {
  id: string
  type: 'door' | 'window'
  offsetMm: number    // от x1,y1 линии
  widthMm: number
  heightMm: number
  sillHeightMm?: number
  label: string
}

export interface PlanLine {
  id: string
  x1: number; y1: number; x2: number; y2: number
  type: PlanLineType
  lengthMm: number
  label: string
  spec?: PlanLineSpec
  wallId?: string; liningId?: string
  openings?: PlanOpening[]
  heightMm?: number
  category?: LineCategory   // капитал (периметр/колонны) или изменяемая
  workStatus?: WorkStatus   // статус работ — актуально для mutable
}

export interface BackgroundImage {
  dataUrl: string
  x: number; y: number
  width: number; height: number
  opacity: number
  locked: boolean
}

export interface Room {
  id: string
  lineIds: string[]
  areaM2: number
  perimeterMm: number
  label: string
  templateName?: string
  isColumn?: boolean   // true — это колонна (штриховка, скрыт периметр в подписи)
}

export interface FloorPlan {
  scaleMmPerPx: number
  lines: PlanLine[]
  contours: PlanContour[]
  rooms: Room[]
  backgroundImage?: BackgroundImage | null
}
```

`WallForJoin` (core/wallJoin.ts) — расчётная структура, не хранится в
сторе, строится на лету перед `computeWallJoins()`:
```typescript
export interface WallForJoin {
  id: string
  x1: number; y1: number; x2: number; y2: number
  halfPx: number
  createdIndex: number
  category?: 'capital' | 'mutable'  // слой 2 — приоритет в T-стыках
}
```

## Известные баги/долги (актуализировано)

1. getStatus() — заглушка, всегда 'none' (не путать с новым `workStatus`
   на линиях — это разные вещи, старая заглушка ещё не убрана из UI таблицы)
2. Кнопка "Открыть полный расчёт" — пустой колбэк
3. Кнопки "Дублировать" (⧉) и "Меню" (⋮) — без обработчиков
4. Крайние стойки торчат за периметр на canvas
5. Проёмы (offsetMm) всё ещё от x1,y1 линии, не проверено на митровых углах
6. Упаковки саморезов: TN/MN/XTN→1000, LN→500
7. Проёмы визуально — только разрыв+подпись, без символа открывания (дуга) —
   сознательное решение "пока не нужно"
8. **Правая колонка-инспектор пустая, когда линия/помещение не выбраны** —
   обсуждали идею постоянной сводки по объекту, решение не принято
9. Плана→реальный расчёт нет — только площадь (calcLineArea), не раскрой/
   саморезы. См. "Переводчик" в роадмапе выше — главная незакрытая задача

## СЛЕДУЮЩАЯ ЗАДАЧА (открыто, не начато)

Пользователь уходит на работу, будет пробовать погонять план дальше и
искать баги на реальных объектах — это, судя по этому чату, продуктивный
способ находить настоящие проблемы (за сессию так нашли и поправили два
реальных бага: T-стык по толщине стены, толщина превью-линии).

Варианты на начало следующего чата:
- Если пользователь принёс новые баги/скриншоты с реального использования —
  разбирать их первыми
- Иначе — переводчик `PlanLine → SurfaceSheetInput` (слой 4), самая
  ценная незакрытая задача: даст реальный расчёт (не только площадь) для
  ЛЮБОЙ линии плана, включая замкнутый контур колонны одним проходом
- Слой 3 (типовые колонны, библиотека) — после переводчика имеет смысл
  больше, т.к. тогда колонна сразу и считается, и вставляется одним кликом

Спросить пользователя в начале следующего чата, что в приоритете /
что нашёл на реальном объекте.

## Стратегический контекст (из разговора этой сессии, не код)

Пользователь честно сформулировал долгосрочное видение: wall-calc должен
вырасти в «конструктор объекта в реальном времени» — калька с PDF → статусы
работ → 3D-вид, где здание рисуется как RTS-стратегия (юниты=конструкции,
ресурсы=материалы, статус=прогресс стройки). Идея принята как верная
конечная цель, но осознанно НЕ реализуется одним рывком. Согласован порядок
слоёв:

1. **Материал примыкания + резолвер крепежа** — ✅ сделано этой сессией
2. **Переводчик PlanLine → SurfaceSheetInput** (раскрой) — ✅ сделано этой сессией (только раскрой, без профиля/крепежа профиля)
3. Разрез пол/потолок (балки, уклоны, ступени) как визуальный редактор поверх уже готовой `EdgeProfile`-математики — не начато
4. 3D-вид — не начато, в проекте нет ни одной 3D-библиотеки

## Что сделано в этом чате (хронологически)

### 1. attachmentResolver.ts — геометрия бокового примыкания
Новый модуль. Отвечает на вопрос "к чему физически прикручен конец линии" —
не рисует, только определяет факт примыкания (T или L, та же логика допуска,
что в `wallJoin.ts`) и материал соседа.

- `AttachSurface`, `EndAttachment`, `LineAttachments` — типы
- `resolveLineAttachments(lineId, surfaces)` / `resolveAllAttachments(surfaces)`
- `attachmentMaterialOf(type, specMaterial)`: `wall_existing` → brick/block/
  concrete/unknown по `spec.material`; `wall_new`/`wall_lining` → всегда
  `gkl_existing` (для соседней стойки неважно "новая" она или "существующая"
  в системе статусов — физически это ГКЛ)
- Подключён в `FloorPlan.tsx`: `lineAttachments` useMemo строит
  `AttachSurface[]` из `lines` (аналогично уже существовавшему `wallJoins`)

### 2. fastenerCatalog.ts + calcAttachmentFasteners.ts — крепёж примыкания
Пользователь дал практические дефолты (не таблица Кнауф, а то, что реально
используется на объекте):
- Бетон/кирпич → дюбель 6×40
- Газо-/пеноблок → саморез по дереву 45 (55 — как опция)
- ГКЛ → существующая ГКЛ-конструкция → саморез по металлу (дюбель-бабочка
  формально рекомендуется Кнауф, но на практике почти не используется —
  есть в списке, не дефолт)
- Анкер-клин 6×40 / просечка / кровельный саморез — в каталоге как опции
  для других сценариев (подвесы в потолок, металл разной толщины),
  но пока не резолвятся автоматически — только ручной выбор

`suggestFastener(material)` — дефолт по материалу. `FASTENER_OPTIONS`,
`ATTACHMENT_MATERIAL_LABEL` — для UI. Ручной override хранится на
`PlanLine.fastenerStart`/`fastenerEnd` (побеждает дефолт).

`calcAttachmentFasteners.ts`: `resolveEndFastener`, `fastenerCountByHeight`
(конвенция `Math.ceil(h/step)`, без "+1" — как в `screwsByHeight` в
`calcScrews.ts`, для единообразия по всему проекту), `calcLineFasteners`,
`calcProjectFasteners` (агрегация по типу для всего плана — пока никуда
не выведена, задел под будущую смету).

UI: инспектор линии (только `wall_new`/`wall_lining`) — блок «Боковое
примыкание и крепёж», по одному под-блоку на конец линии: материал соседа
(или «свободный край»), дропдаун типа крепежа (предзаполнен дефолтом),
поле шага в мм, живое количество «N шт».

### 3. Шаг стоек и лист обшивки на PlanLineSpec
Обнаружен пробел: `PlanLineSpec` не хранил `step`/`layer1`/`layer2` — без
них невозможен переводчик в `SurfaceSheetInput`. Решение (согласовано
с пользователем, "вариант 3" — гибрид): глобальный дефолт при рисовании
(как уже было с `drawHeightMm`) + точечное переопределение в инспекторе.

- `PlanLineSpec.step?`, `.layer1?`, `.layer2?` (`BoardSpec`)
- Левая панель: поле «Шаг стоек» + `BoardSpecSelector` (переиспользован
  существующий компонент) для листа 1-го/2-го слоя, рядом с «Высота»
- Инспектор: аналогичный блок-переопределение для конкретной линии
  (только `wall_new`/`wall_lining` с `material === 'gkl'`)
- **Известный косметический минус:** `BoardSpecSelector` стилизован под
  светлый фон (белые дропдауны), а левая панель тёмная — визуально не
  вписывается, но функционально работает. Не полировалось.

### 4. planLineToSurfaceInput.ts — переводчик, слой 4 (раскрой)
Перед реализацией выяснили с пользователем важный факт по системам
облицовки (пригодится для СЛЕДУЮЩЕГО переводчика, не для этого):

- **С623**: направляющая ПН/ПС 27×28, стойка ПП 60×27, + прямой подвес
  (крепёж к стене) всегда, + удлинитель если высота > 3000мм. Слои 1 или 2.
  Соответствует `spec.subtype === 'frame_pn28'`.
- **С625**: профиль ПС 50×50 / 75×50 / 100×50, один слой.
  `frame_ps50`/`frame_ps75` при `layers === 1`.
- **С626**: те же профили ПС, но 2 слоя обшивки.
  `frame_ps50`/`frame_ps75` при `layers === 2`.
- Отличаются требованиями шага и ограничением по высоте в зависимости
  от профиля (см. `liningMaxHeight.ts` — уже существует для standalone-формы).
- Облицовка на клею — это **С611**, отдельный тип. Расчёта по ней **нет**,
  пользователь подтвердил, что клеевой расход никогда не считали.
  `spec.subtype === 'glued'` — калькулятора материала под это пока нет.
- ⚠️ Нет `frame_ps100` в дереве материалов `wall_lining` (только
  `frame_ps50`/`frame_ps75`) — пробел, если понадобится ПС100 для облицовки.
- ⚠️ `PlanLineSpec` не хранит `hangerStep` (шаг прямых подвесов) — ещё
  один пробел для будущего переводчика в сторону профиля/подвесов.

**Ключевая находка, упростившая задачу:** раскрой листов (`calcSheetLayout`)
не зависит от `liningType`/`profileType` вообще — только от геометрии
(длина/высота/проёмы), шага стоек и спецификации листа. Вся вышеописанная
классификация С623/625/626 нужна только для БУДУЩЕГО переводчика в сторону
профиля/крепежа/подвесов, не для этого слоя. Переводчик написан без неё:

- `planLineToSurfaceInput(line)`: `null` для `wall_existing`, кладки
  (brick/gasblock/foamblock на wall_new), плитки/штукатурки/малярки —
  всё, что не является ГКЛ-раскроем
- `sides`: `wall_new` → 2 (обе стороны перегородки), `wall_lining` → 1
- Упрощение (как и в существующем `buildSurfaceInputs` для облицовки):
  позиций стоек на линии плана нет (не запускается полный
  `calcStudMaterial`), поэтому `firstStud = step`. Высота — плоская
  (`heightMm ?? 3000`), без уклонов/ступеней (это пункт 3 дорожной карты)
- `planLinesToSurfaceInputs(lines)` — батч, сохраняет порядок, отбрасывает null
- **Ещё не подключено к UI** — результат `calcProjectSheetLayout(planLinesToSurfaceInputs(lines))`
  можно посчитать хоть сейчас, но негде показать (нет вкладки/кнопки сметы
  на плане)

## Известные особенности / договорённости для будущих сессий

(пункты 1-6 из предыдущего конспекта по T-снапу/подложке/проёмам остаются
в силе, не повторяю — см. git-историю FloorPlan.tsx при необходимости)

7. **Резолверы `wallJoin`/`attachmentResolver` — оба O(n²)** (каждая линия
   против каждой). На типичных планах не проблема, но если объект вырастет
   до многих сотен линий и появятся жалобы на тормоза при рисовании —
   проверять первым делом это.

8. **Мэппинг subtype → liningType для облицовки** (см. пункт 4 выше) уже
   выяснен у пользователя и записан здесь — не переспрашивать в следующий
   раз, сразу использовать при постройке переводчика в сторону профиля.

9. **`hangerStep` на PlanLineSpec отсутствует.** Нужен для C623 (прямые
   подвесы — обязательны) и в целом для расчёта подвесов облицовки.
   Дефолт в standalone-форме — 1000мм (`LiningCalc.tsx`). Добавлять по
   тому же паттерну глобал+override, что и `step`/`layer1`/`layer2`.

10. **`frame_ps100` отсутствует в дереве материалов `wall_lining`.**
    Если понадобится ПС100 для облицовки — добавить в `constructionTaxonomy.ts`.

11. **Крепёж (`FastenerType`) резолвится и считается по количеству, но
    нигде не суммируется в общую смету.** `calcProjectFasteners()` готова,
    просто не вызывается из UI.

12. **`suggestFastener()` не покрывает все сценарии из каталога** —
    анкер-клин (подвесы в потолок) и просечка/кровельный саморез (металл
    по толщине) есть в `FASTENER_OPTIONS`, но не в `suggestFastener()`,
    потому что это другие сценарии крепления (не боковое примыкание стойки),
    сейчас доступны только ручным выбором в дропдауне.

## Архитектура данных (types/index.ts) — новое в этой сессии

```typescript
export type AttachmentMaterial = 'brick' | 'block' | 'concrete' | 'gkl_existing' | 'unknown'

export type FastenerType =
  | 'dowel_6x40' | 'wood_screw_45' | 'wood_screw_55' | 'metal_screw'
  | 'gypsum_toggle' | 'anchor_wedge_6x40' | 'self_drill_screw' | 'roofing_screw'

export interface FastenerSpec {
  type: FastenerType
  stepMm: number
}

export interface PlanLineSpec {
  material: string
  subtype?: string
  boardSubtype?: GklSubtype
  layers?: 1 | 2
  step?: number          // НОВОЕ: шаг стоек, мм
  layer1?: BoardSpec      // НОВОЕ: спецификация листа 1-го слоя
  layer2?: BoardSpec      // НОВОЕ: спецификация листа 2-го слоя
}

export interface PlanLine {
  // ...существующие поля...
  fastenerStart?: FastenerSpec  // НОВОЕ: ручное переопределение крепежа, конец (x1,y1)
  fastenerEnd?: FastenerSpec    // НОВОЕ: ручное переопределение крепежа, конец (x2,y2)
}
```

## ДОЛГОСРОЧНОЕ ВИДЕНИЕ ПРОЕКТА (зафиксировано 03.07.2026, НЕ в работе)

Пользователь (монтажник/прораб, не программист — держать объяснения
простыми в диалоге с ним) сформулировал развёрнутую конечную цель.
Сознательно отложено ради текущих задач (раскрой/материалы/баги),
но должно быть на виду для планирования порядка следующих слоёв.

**Идея одной фразой:** RTS-стратегия, перенесённая на реальную стройку.
Холст — не чертёж, а живая модель объекта. Каждый объект на плане
(стена, перегородка, плитка, потолок) — не линия, а сущность со
своими свойствами и статусом. Ресурсы (материалы) считаются сразу,
как только объект появился на холсте (это уже частично работает —
расчётный движок готов, просто не для всех типов объектов и не
привязан к статусам).

**Статусы объекта** (сейчас `getStatus()` — заглушка, всегда 'none'):
- существующий (стоит и было до нас)
- неактивный (запланирован, работа не началась)
- в работе
- закончен
- заблокирован — с причиной (нет материала / ждём смежников / другое).
  Причина блока — отдельное поле, не просто ярлык, пригодится для
  списка "что сейчас держит стройку".

**3D-вид, реагирующий на статус в реальном времени.** Меняешь статус
линии на "в работе" — на вкладке 3D конструкция начинает возводиться
прямо на глазах. "Существующие" конструкции в 3D уже стоят изначально.
Технически: 3D-сцена строится из тех же данных, что и холст (материал,
высота, проёмы — всё уже есть в моделе `PlanLine`), просто нужен
рендер-слой поверх них. **В проекте пока нет НИ ОДНОЙ 3D-библиотеки** —
это будет отдельный крупный кусок работы, не мелкая фича.

**Совместная работа в реальном времени.** "Все, кто участвует в работе
над проектом" видят изменения статусов друг друга сразу. База есть
(Supabase, уже используется для авторизации/хранения проектов), но
сам realtime-канал (подписка на изменения) пока не подключён.

**Шаблоны однотипных квартир.** Если фирма делает ремонт в одинаковых
квартирах — не перерисовывать план заново на каждую, а иметь шаблон,
"проштамповать" его N раз с нумерацией, и в 3D эти копии встают друг
на друга как этажи. Логически независимо от статусов/3D-движка,
может делаться параллельно или отдельной веткой позже.

**Согласованный порядок (по итогам разговора):** статусы — это
фундамент, без него не имеет смысла начинать 3D (нечего будет туда
передавать кроме голой геометрии). 3D-движок — самый затратный по
времени слой, начинать точно не сейчас. Шаблоны квартир — не
зависят от первых двух, можно рассмотреть отдельно, если появится
практическая нужда раньше.

**Что спросить в начале сессии, если разговор зайдёт про это снова:**
готовы ли наконец взяться за статусы (это открывает путь к 3D), или
всё ещё приоритет — текущий расчётный функционал.

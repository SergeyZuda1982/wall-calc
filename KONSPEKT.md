## Что сделано в сессии 04.07.2026 (эта сессия — 3D, этажи, карандаш)

Пользователь — прораб/монтажник, не программист, работает интуитивно,
предпочитает простой язык без терминов. Держать объяснения простыми.

### 1. Ригель (rib_beam) — новая сущность на плане
Балка перекрытия между колоннами (иногда крест-накрест, иногда
параллельно), физически единое целое с плитой перекрытия сверху —
резать нельзя, коммуникации в 3D должны идти в обход (сама логика
обхода — НЕ сделана, это просто геометрический факт зафиксирован).
Высота чистового потолка под ней ограничена её "опусканием" от плиты.

- `PlanLineType.rib_beam`, `PlanLine.sectionWidthMm`/`.dropMm` (своя
  геометрия, НЕ материальная taxonomy — `TAXONOMY.rib_beam = []`)
- Всегда `category:'capital'`, `workStatus:'existing'` по умолчанию
  (как периметр/колонны)
- Рисуется как обычная линия (клик-клик), глобальные дефолты сечения/
  опускания в левой панели
- Автоматически исключён из `wallJoin`/`attachmentResolver` (оба читают
  `hasSpec = !!spec.material`, у ригеля spec не задаётся)
- Обшивка ГКЛ на ригеле — НЕ отдельная сущность, рисуется обычной
  `wall_lining` линией рядом, как уже делается для колонн

### 2. Первая 3D-сцена объекта — **впервые в проекте появилась 3D-библиотека**
Установлены `three`, `@react-three/fiber`, `@react-three/drei`.
Новая вкладка **«🧊 3D»** в `App.tsx`, статичный вид всего, что на плане.

- `src/core/planTo3D.ts` — ЧИСТАЯ математика (без импорта three.js,
  тестируется без браузера/WebGL): `wallToBox3D` (короб на линию —
  толщина/высота/поворот вокруг Y), `estimateCeilingMm` (высота потолка
  = максимум heightMm среди wall_existing, или дефолт 3000 — это ЗАМЕНА
  настоящей высоты потолка, её как отдельной величины в модели нет),
  `roomsToPolygons3D` (контуры помещений/колонн в метры),
  `slabsToPolygons3D` (см. п.5 ниже). 19 тестов в этом файле.
- `src/core/contour.ts` — `extractContourPoints` вынесена из
  `FloorPlan.tsx`, переиспользуется и в 2D, и в 3D
- `src/Scene3D.tsx` — сборка three.js-мешей (`boxGeometry` для стен/
  ригелей, `THREE.Shape`+`ExtrudeGeometry` по контуру для пола/потолка/
  колонн), `OrbitControls`
- **Сознательные упрощения v1** (не сделано, известно):
  - проёмы (двери/окна) НЕ вырезаются из объёма стены — сплошной короб
  - митры/стыки стен (`wallJoin`) не учитываются в 3D — короба просто
    накладываются друг на друга по углам
  - нет анимации/фильтра по `workStatus` — показывается вся геометрия
    как есть, статусы (уже есть в данных, см. ниже) пока не влияют на 3D
  - показывается только ОДИН этаж (активный), не все разом

### 3. Режим "полёта" в 3D (WASD + мышь)
Мышь-только (`OrbitControls`) была неудобной. `FlyControls` из drei,
переключатель в углу канваса. `dragToLook` — камера поворачивается
только при зажатой мыши, WASD двигают всегда. R/F — вверх/вниз.

**Инцидент на практике:** деплой одного из коммитов упал с "Deployment
failed, try again later" (не связано с кодом — временный сбой GitHub
Pages). Вылечено пустым коммитом-триггером. Отсюда правило 6 выше.

### 4. Многоэтажность — шаг 1 (данные + минимальный переключатель)
Пользователь был прорабом и на 5-этажном офисе — решили не ограничиваться
одним этажом сразу, закладывать многоэтажность с фундамента.

- `types/index.ts`: `Level { id, name, elevationMm, floorPlan }`,
  `emptyLevel()`. `elevationMm` — отметка НИЗА этажа от условного нуля.
- `store/useProjectStore.ts` — **рефактор структуры данных**:
  `ProjectEntry.levels: Level[]` + `activeLevelId` ВМЕСТО одного
  `floorPlan`. Все существующие действия (`addPlanLine`, `updateRoom`
  и т.п.) переведены на общий helper `updateActiveFloorPlan()` — пишут
  в план АКТИВНОГО этажа. `store.floorPlan` остался плоским зеркалом
  (= план активного этажа) — поэтому `FloorPlan.tsx`/`Scene3D.tsx`
  почти не пришлось трогать, они как и раньше читают `store.floorPlan`.
  Новые действия: `addLevel`, `duplicateLevel`, `removeLevel`,
  `renameLevel`, `setLevelElevation`, `selectLevel`.
  Миграция в `onRehydrateStorage`: старые сохранённые проекты (floorPlan
  прямо на объекте, без levels) оборачиваются в один "Этаж 1" — на
  практике сохранённых объектов сейчас нет, но на будущее не лишнее.
- `FloorPlan.tsx`: панель этажей в шапке плана — переключить/добавить/
  дублировать/удалить/переименовать/поменять отметку. **Через
  `window.prompt`, без красивых форм** — сознательно, чтобы сначала
  проверить механику, эстетика — следующий шаг ("сначала карандаш,
  потом причесать интерфейс", так решил пользователь).
- **Подтверждено пользователем на живом сайте — механика работает.**

### 5. Инструмент "карандаш" — плита + вырезание проёмов
Идея пользователя: обвести периметр здания произвольным контуром
(не привязан к прямым стенам, в отличие от `Room`) → получить плиту
пола/потолка. Тем же инструментом — вырезать дырки под лестницы/шахты/
стволы коммуникаций (коммуникации, обходящие дырки — НЕ сделано, только
геометрия дырки существует). "Дублировать" плиту на другую отметку —
задел на будущее (см. пробелы ниже), сейчас плита живёт внутри одного
этажа, копировать между этажами вручную пока негде.

- `types/index.ts`: `Slab { id, outer, holes, label }`. Своего поля
  высоты у плиты НЕТ — отметка берётся с этажа (`Level.elevationMm`),
  плита = пол текущего этажа. `FloorPlan.slabs: Slab[]`.
- `store`: `addSlab`/`updateSlabOuter`/`addSlabHole`/`removeSlab`/
  `removeSlabHole`, через тот же `updateActiveFloorPlan`.
- `FloorPlan.tsx`: режим `'pencil'` — клик добавляет точку контура,
  клик рядом с первой точкой замыкает (тот же порог, что у смыкания
  цепочки стен), ПКМ/Esc отменяет текущий контур. Список плит в левой
  панели — клик по плите переключает в режим "следующий контур — дырка
  в НЕЙ". Рендер — Konva `Shape` с evenodd-заливкой (`moveTo`/`lineTo`/
  `closePath` на внешнем контуре и каждой дырке).
- `core/planTo3D.ts`: `slabsToPolygons3D()` — контур+дырки в метры.
- `Scene3D.tsx`: `HandDrawnSlabMesh` — `THREE.Shape` + `Shape.holes`
  (`THREE.Path`), рендерится как пол (y=0). Если для этажа нарисованы
  плиты карандашом — старый авто-пол из `Room`-контуров отключается
  (`skipFloor`), чтобы не задваивалось; авто-потолок из `Room` оставлен
  как единственный источник потолка (плита пока не даёт автоматический
  потолок — это будет через плиту этажа ВЫШЕ, когда 3D научится
  показывать несколько этажей разом).
- **Не удалить плиту/дырку из UI** — действия в сторе есть, кнопок нет
  (эстетика — следующий шаг).
- Смержено в main **с настоящим merge-конфликтом** — параллельный чат
  (`feat/sealing-tape`) в то же время трогал тот же массив зависимостей
  `useCallback` в `FloorPlan.tsx`. Разрешено вручную (объединены оба
  набора зависимостей). Первый реальный конфликт за всю практику
  ветвления в этом проекте — подтверждает, что правило "fetch+reset
  перед мержем" не паранойя, а необходимость.

### Обсуждённая, но НЕ реализованная идея — односторонняя прозрачность
Пользователь описал желаемое поведение плиты в 3D: пол/потолок должны
быть непрозрачными (серый бетон до отделки) со "своей" стороны (где
стоит человек) и прозрачными с обратной — чтобы при облёте здания сверху
можно было смотреть сквозь этажи вглубь (dollhouse-вид), а находясь
внутри комнаты — видеть сплошной потолок/пол. Технически это стандартный
приём three.js (материал с `side: THREE.FrontSide`, односторонний рендер
грани) — не сложно, но НЕ сделано, отложено до многоэтажного 3D.

## Известные пробелы именно по 3D/этажам/карандашу (открыто)

1. **Проёмы не вырезаны из стен в 3D** — сплошной короб, дверь/окно
   на плане есть, в 3D нет.
2. **Митры стен (`wallJoin`) не учтены в 3D** — короба перекрываются
   по углам "как есть", визуально заметно на острых углах.
3. **3D показывает только один (активный) этаж** — нет совместного
   вида всех этажей разом с учётом их отметок. Это нужно и для
   правильных потолков (плита этажа N = потолок этажа N-1).
4. **Односторонняя прозрачность пола/потолка** — обсуждена, не сделана
   (см. выше).
5. **Дублирование ПЛИТЫ на другой этаж** — пользователь хотел именно
   это ("нарисовал пол, дублировал, задал новую отметку — получил
   потолок/пол следующего этажа"), а не дублирование ЦЕЛОГО этажа
   (это уже есть, `duplicateLevel`, но копирует вообще всё, не только
   плиту). Не сделано.
6. **Удаление плиты/дырки из UI** — только через store-действия, кнопок нет.
7. **Интерфейс этажей** — `window.prompt`, не формы. Пользователь
   осознанно попросил делать в этом порядке: сначала карандаш, ПОТОМ
   причесать интерфейс — это следующая согласованная задача.
8. **Коммуникации не обходят дырки в плите** — только геометрия дырки
   существует, логики "труба видит дырку и не проходит сквозь плиту
   мимо неё" нет.
9. Бандл сильно подрос из-за three.js (~2.1MB минифицированного JS) —
   код-сплиттинг не сделан, если станет проблемой при загрузке — делать
   отдельным шагом (`vite` уже предупреждает об этом при сборке).

## Архитектура данных — новое за эту сессию (сверху существующего)

```typescript
// PlanLineType — добавлен rib_beam
export type PlanLineType =
  | 'wall_new' | 'wall_lining' | 'wall_existing' | 'ceiling' | 'floor'
  | 'rib_beam'   // ригель — балка перекрытия, монолит, режется нельзя

export interface PlanLine {
  // ...существующие поля (spec, category, workStatus, openings, heightMm...)
  sectionWidthMm?: number   // rib_beam: ширина сечения по плану, мм
  dropMm?: number           // rib_beam: опускание от плиты перекрытия, мм
}

// Этаж — НОВОЕ, заменяет собой одиночный floorPlan на проекте
export interface Level {
  id: string
  name: string          // "Этаж 1", "Цоколь"...
  elevationMm: number    // отметка НИЗА этажа от условного нуля объекта
  floorPlan: FloorPlan
}

// Плита ("карандаш") — НЕ привязана к стенам, в отличие от Room
export interface Slab {
  id: string
  outer: { x: number; y: number }[]      // внешний контур, px
  holes: { x: number; y: number }[][]    // вырезы (лестницы/шахты)
  label: string
}

export interface FloorPlan {
  scaleMmPerPx: number
  lines: PlanLine[]
  contours: PlanContour[]
  rooms: Room[]
  slabs: Slab[]         // НОВОЕ
  backgroundImage?: BackgroundImage | null
}
```

`ProjectEntry` (в сторе): `levels: Level[]` + `activeLevelId: string`
ВМЕСТО `floorPlan: FloorPlan` (было раньше). Top-level `store.floorPlan`
остался как зеркало плана активного этажа — большинство кода-потребителя
(`FloorPlan.tsx`, `Scene3D.tsx`) этого не заметило.

## СЛЕДУЮЩАЯ ЗАДАЧА (согласовано с пользователем на конец сессии)

Порядок, который пользователь явно попросил: **сначала карандаш (плита),
потом интерфейс** — карандаш сделан и подтверждён на сайте, значит
следующий шаг — **полировка интерфейса**:
1. Интерфейс этажей — заменить `window.prompt` на нормальные формы/панель.
2. Интерфейс плит/дырок — кнопки удаления, список с понятными подписями.
3. После интерфейса — открытые пробелы по приоритету, который решит
   пользователь: дублирование плиты на другой этаж, многоэтажный 3D-вид
   с отметками, односторонняя прозрачность, проёмы/митры в 3D, либо

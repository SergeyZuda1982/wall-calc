## Стратегический контекст (из разговора этой сессии, не код)

Пользователь честно сформулировал долгосрочное видение: wall-calc должен
вырасти в «конструктор объекта в реальном времени» — калька с PDF → статусы
работ → 3D-вид, где здание рисуется как RTS-стратегия (юниты=конструкции,
ресурсы=материалы, статус=прогресс стройки). Идея принята как верная
конечная цель, но осознанно НЕ реализуется одним рывком. Согласован порядок
слоёв:

1. **Материал примыкания + резолвер крепежа** — ✅ сделано этой сессией
2. **Переводчик PlanLine → SurfaceSheetInput** (раскрой) — ✅ сделано этой сессией (только раскрой, без профиля/крепежа профиля)
3. Разрез пол/потолок (балки, уклоны, ступени) как визуальный редактор поверх уже готовой `EdgeProfile`-математики — не начато
4. 3D-вид — не начато, в проекте нет ни одной 3D-библиотеки

## Что сделано в этом чате (хронологически)

### 1. attachmentResolver.ts — геометрия бокового примыкания
Новый модуль. Отвечает на вопрос "к чему физически прикручен конец линии" —
не рисует, только определяет факт примыкания (T или L, та же логика допуска,
что в `wallJoin.ts`) и материал соседа.

- `AttachSurface`, `EndAttachment`, `LineAttachments` — типы
- `resolveLineAttachments(lineId, surfaces)` / `resolveAllAttachments(surfaces)`
- `attachmentMaterialOf(type, specMaterial)`: `wall_existing` → brick/block/
  concrete/unknown по `spec.material`; `wall_new`/`wall_lining` → всегда
  `gkl_existing` (для соседней стойки неважно "новая" она или "существующая"
  в системе статусов — физически это ГКЛ)
- Подключён в `FloorPlan.tsx`: `lineAttachments` useMemo строит
  `AttachSurface[]` из `lines` (аналогично уже существовавшему `wallJoins`)

### 2. fastenerCatalog.ts + calcAttachmentFasteners.ts — крепёж примыкания
Пользователь дал практические дефолты (не таблица Кнауф, а то, что реально
используется на объекте):
- Бетон/кирпич → дюбель 6×40
- Газо-/пеноблок → саморез по дереву 45 (55 — как опция)
- ГКЛ → существующая ГКЛ-конструкция → саморез по металлу (дюбель-бабочка
  формально рекомендуется Кнауф, но на практике почти не используется —
  есть в списке, не дефолт)
- Анкер-клин 6×40 / просечка / кровельный саморез — в каталоге как опции
  для других сценариев (подвесы в потолок, металл разной толщины),
  но пока не резолвятся автоматически — только ручной выбор

`suggestFastener(material)` — дефолт по материалу. `FASTENER_OPTIONS`,
`ATTACHMENT_MATERIAL_LABEL` — для UI. Ручной override хранится на
`PlanLine.fastenerStart`/`fastenerEnd` (побеждает дефолт).

`calcAttachmentFasteners.ts`: `resolveEndFastener`, `fastenerCountByHeight`
(конвенция `Math.ceil(h/step)`, без "+1" — как в `screwsByHeight` в
`calcScrews.ts`, для единообразия по всему проекту), `calcLineFasteners`,
`calcProjectFasteners` (агрегация по типу для всего плана — пока никуда
не выведена, задел под будущую смету).

UI: инспектор линии (только `wall_new`/`wall_lining`) — блок «Боковое
примыкание и крепёж», по одному под-блоку на конец линии: материал соседа
(или «свободный край»), дропдаун типа крепежа (предзаполнен дефолтом),
поле шага в мм, живое количество «N шт».

### 3. Шаг стоек и лист обшивки на PlanLineSpec
Обнаружен пробел: `PlanLineSpec` не хранил `step`/`layer1`/`layer2` — без
них невозможен переводчик в `SurfaceSheetInput`. Решение (согласовано
с пользователем, "вариант 3" — гибрид): глобальный дефолт при рисовании
(как уже было с `drawHeightMm`) + точечное переопределение в инспекторе.

- `PlanLineSpec.step?`, `.layer1?`, `.layer2?` (`BoardSpec`)
- Левая панель: поле «Шаг стоек» + `BoardSpecSelector` (переиспользован
  существующий компонент) для листа 1-го/2-го слоя, рядом с «Высота»
- Инспектор: аналогичный блок-переопределение для конкретной линии
  (только `wall_new`/`wall_lining` с `material === 'gkl'`)
- **Известный косметический минус:** `BoardSpecSelector` стилизован под
  светлый фон (белые дропдауны), а левая панель тёмная — визуально не
  вписывается, но функционально работает. Не полировалось.

### 4. planLineToSurfaceInput.ts — переводчик, слой 4 (раскрой)
Перед реализацией выяснили с пользователем важный факт по системам
облицовки (пригодится для СЛЕДУЮЩЕГО переводчика, не для этого):

- **С623**: направляющая ПН/ПС 27×28, стойка ПП 60×27, + прямой подвес
  (крепёж к стене) всегда, + удлинитель если высота > 3000мм. Слои 1 или 2.
  Соответствует `spec.subtype === 'frame_pn28'`.
- **С625**: профиль ПС 50×50 / 75×50 / 100×50, один слой.
  `frame_ps50`/`frame_ps75` при `layers === 1`.
- **С626**: те же профили ПС, но 2 слоя обшивки.
  `frame_ps50`/`frame_ps75` при `layers === 2`.
- Отличаются требованиями шага и ограничением по высоте в зависимости
  от профиля (см. `liningMaxHeight.ts` — уже существует для standalone-формы).
- Облицовка на клею — это **С611**, отдельный тип. Расчёта по ней **нет**,
  пользователь подтвердил, что клеевой расход никогда не считали.
  `spec.subtype === 'glued'` — калькулятора материала под это пока нет.
- ⚠️ Нет `frame_ps100` в дереве материалов `wall_lining` (только
  `frame_ps50`/`frame_ps75`) — пробел, если понадобится ПС100 для облицовки.
- ⚠️ `PlanLineSpec` не хранит `hangerStep` (шаг прямых подвесов) — ещё
  один пробел для будущего переводчика в сторону профиля/подвесов.

**Ключевая находка, упростившая задачу:** раскрой листов (`calcSheetLayout`)
не зависит от `liningType`/`profileType` вообще — только от геометрии
(длина/высота/проёмы), шага стоек и спецификации листа. Вся вышеописанная
классификация С623/625/626 нужна только для БУДУЩЕГО переводчика в сторону
профиля/крепежа/подвесов, не для этого слоя. Переводчик написан без неё:

- `planLineToSurfaceInput(line)`: `null` для `wall_existing`, кладки
  (brick/gasblock/foamblock на wall_new), плитки/штукатурки/малярки —
  всё, что не является ГКЛ-раскроем
- `sides`: `wall_new` → 2 (обе стороны перегородки), `wall_lining` → 1
- Упрощение (как и в существующем `buildSurfaceInputs` для облицовки):
  позиций стоек на линии плана нет (не запускается полный
  `calcStudMaterial`), поэтому `firstStud = step`. Высота — плоская
  (`heightMm ?? 3000`), без уклонов/ступеней (это пункт 3 дорожной карты)
- `planLinesToSurfaceInputs(lines)` — батч, сохраняет порядок, отбрасывает null
- **Ещё не подключено к UI** — результат `calcProjectSheetLayout(planLinesToSurfaceInputs(lines))`
  можно посчитать хоть сейчас, но негде показать (нет вкладки/кнопки сметы
  на плане)

## Фича 03.07.2026: ввод дуги напрямую по радиусу R

Прямое продолжение фичи дуг из этой же сессии. Пользователь объяснил
конкретную причину с реального объекта (фото арок из этого же чата):
несколько арок между разными колоннами (разный пролёт/хорда), но
ЗАДАЛ одну и ту же стрелу H на каждой — из-за разной хорды у них
получился разный, никак не связанный друг с другом радиус. Правильный
путь — наоборот: один и тот же R, а H на каждом пролёте своя.

**Геометрия — `geometry2d.ts` → `sagittaFromRadius(chordLen, radius, deep)`:**
обратная задача к `arcFromChordAndSagitta` (та же формула R=(L²+H²)/2H,
решённая относительно H). Важный нюанс, из-за которого добавлен параметр
`deep`: при R больше половины хорды решения ДВА — пологая дуга
(H=R−√(R²−L²), обычный архитектурный случай) и глубокая (H=R+√(R²−L²),
больше полуокружности). При R меньше половины хорды решений нет вообще
(хорда физически не влезает в окружность такого маленького радиуса) —
возвращает `null`, вызывающий код обязан это проверять, не считать
результат всегда числом. 6 новых тестов, включая round-trip (H→R→H в
обе стороны) и ключевой практический случай (один R, разные пролёты →
разные H, но одинаковый R при проверке обратно).

**UI рисования:** переключатель "Дуга по: H | R" в левой панели.
В режиме R — поле радиуса + чекбокс "глубокая дуга". Стрела H
вычисляется в момент завершения линии (`computeSagittaForNewLine` в
`FloorPlan.tsx`), когда хорда уже известна по факту клика — до этого
момента R сам по себе не определяет H (нужна ещё и хорда). Если R
меньше минимально возможного для нарисованной хорды — `window.alert`
с понятным числом (каким мог бы быть минимальный R), линия остаётся
прямой, а не рисуется что-то неверное молча.

**Инспектор:** оба поля (H и R) видны одновременно рядом, правишь
любое — второе просто отображает то, что получилось. Плюс подсказка
"мин. R для этой хорды" прямо под чекбоксом глубокой дуги, чтобы не
пришлось считать вручную, если ввёл слишком маленький R (там, в отличие
от панели рисования, просто тихо не применяет невалидный ввод — хорда
уже известна заранее, есть время увидеть подсказку до ввода, `alert`
не нужен).

347 тестов всего.

## Фича 03.07.2026: дуги на холсте (арки/гнутые перегородки) — "циркуль"

Пользователь прислал реальные фото объекта (арочный проём, гнутые
радиусные потолочные перегородки) и классическую формулу определения
радиуса дуги по хорде и стреле: R=(L²+H²)/2H, L — половина хорды,
H — стрела (высота подъёма дуги от прямой). Попросил уметь чертить
дуги на холсте, используя в т.ч. эту формулу для ввода.

**Развилка, которую обсудили явно:** просто калькулятор (ввёл L/H,
получил R, без связи с чертежом) vs полноценное умение чертить дуги
на холсте как отдельный тип линии. Выбрано второе — большая работа,
сознательно.

**Данные:** `PlanLine.sagittaMm?: number` — необязательное поле.
Не задано/0 — линия прямая (как раньше, для всех старых линий ничего
не меняется). Задано — линия дуга. `x1/y1/x2/y2` остаются координатами
КОНЦОВ ХОРДЫ (не меняются от того, прямая линия или дуга) — центр
и радиус окружности вычисляются на лету, не хранятся. `lengthMm` для
дуги — это длина ДУГИ, не хорды (используется дальше во всех расчётах
материала как обычно — с этим полем ничего специально готовить не
пришлось, весь стек калькуляции просто получает верное число).

**Геометрия — `geometry2d.ts` → `arcFromChordAndSagitta()`:** строит
центр/радиус/углы/направление обхода по хорде+стреле. Знак sagitta —
сторона выгиба. Работает и для "глубоких" дуг (H > L, больше
полуокружности), не только для пологих. `arcLengthFromSagitta()` —
тонкая обёртка для случаев, когда нужна только длина, без координат
(используется при создании линии и при перетаскивании её конца).
`sampleArcPoints()` — точки вдоль дуги для рендера полилинией
(Konva `Line`, не через canvas `ctx.arc` — решение сознательное, ради
консистентности со стилем остального рендера в `FloorPlan.tsx`, там
везде обычные Konva-компоненты, ни одного `Shape`/`sceneFunc`).
21 новый тест на геометрию (полуокружность и четверть окружности
проверены вручную посчитанными числами — надёжный якорь на будущее).

**Рендер (`FloorPlan.tsx`):** для дуги — отдельная, полностью
самостоятельная ветка в цикле отрисовки линий, ДО проверки `useDouble`
и ДО обращения к `wallJoins`/`computeOpeningSegments`. Заливка стены —
полигон между двумя концентрическими дугами (радиус ±half от
центральной), это ЗНАЧИТЕЛЬНО проще, чем офсет прямой линии — центр
общий, меняется только радиус. Выбор/удаление/подпись/точки-хендлы
концов хорды — переиспользованы один в один с прямой линией (то же
`startDragLine`, тот же `handleLinePointerDown`) — концы хорды это
всё те же `x1/y1/x2/y2`, ничего для этого не пришлось переделывать.

**⚠️ Важный фикс по ходу:** перетаскивание конца линии (`handleMove`,
`dr.kind==='end1'/'end2'`) раньше безусловно писало `lengthMm` как
длину ПРЯМОЙ между новой и старой точкой. Для дуги это было бы тихой
порчей данных при первом же перетаскивании конца. Поправлено — если
`l.sagittaMm` задан, пересчитывает через `arcLengthFromSagitta`.

**Инспектор:** поле "Стрела дуги H" (0 = прямая) + сразу же расчётная
подсказка "Хорда: ... · Радиус R: ..." — тот же расчёт, что и на
объекте (натянул шнур между концами проёма, замерил стрелу посередине
рулеткой, ввёл — увидел R). Работает и на уже нарисованных прямых
линиях (превращает в дугу задним числом) и наоборот (H=0 — снова прямая).

**⚠️ Сознательно НЕ сделано в этом заходе (важно для будущих сессий):**
- **wallJoin/attachmentResolver** — дуга исключена из входа
  `computeWallJoins` (`if (l.sagittaMm) return` в useMemo). Прямая
  стена, упирающаяся в дугу, рисуется без митра/примыкания — просто
  как есть. Стык дуги со стеной — самостоятельная, непростая
  геометрическая задача (не переиспользует ни офсет прямых линий, ни
  обрезку по наклонной), не начата.
- **Проёмы (PlanOpening) на дуге** — не обрабатываются, `openings`
  на дуге просто не рендерятся сейчас (UI не запрещает их добавить,
  это стоит явно проверить/поправить в следующий заход — сейчас не
  проверялось, что произойдёт при попытке).
- **T-снап к телу дуги** — `snapPoint` проецирует точку на ПРЯМОЙ
  отрезок; для дуги это даст неверный результат где-то на теле дуги.
  Снап к КОНЦАМ хорды работает нормально (это обычный endpoint-снап,
  не менялся). Снап к середине/телу дуги — не сделан.
- **Раскрой листов (`calcSheetLayout`) по кривой стене** — не
  адаптирован. `lengthMm` (длина дуги) корректно течёт в расчёт
  материала (это ПРАВИЛЬНО для количества — саморезы/профиль по
  длине), но сама раскладка листов на диаграмме нарисует ПРЯМУЮ стену
  той же длины — визуально неверно для монтажника, но по счёту листов
  скорее всего приемлемо как грубая оценка. Не обсуждалось
  специально, не в скоупе.
- **Превью во время рисования** (пока тянешь второй клик) — рисуется
  ПРЯМОЙ линией, кривизна появляется только после того, как линия уже
  создана. Мелкое упрощение ради экономии времени, не критично —
  пользователь всё равно вводит H заранее, до клика.

**На будущее (сказано пользователем прямо):** скруглённые углы/
окружности как отдельная геометрическая область — НЕ то же самое,
что дуга по хорде+стреле, других переиспользуемых кусков пока нет
(см. комментарий в шапке `geometry2d.ts`).

**⚠️ Стык с 3D, обнаруженный при слиянии этой ветки:** параллельно
(без координации с этой сессией) появилась первая 3D-сцена
(`planTo3D.ts`, `Scene3D.tsx`, three.js/@react-three/fiber — см. раздел
ниже). Она не знает про `sagittaMm` — дуга в 3D отрисуется как прямая
стена по хорде (`x1/y1/x2/y2`), кривизна потеряется. Не чинил —
не в скоупе этой сессии и вне контекста, в котором писался `planTo3D`.
Стоит явно предупредить, когда кто-то возьмётся сводить 3D и дуги.

## Фича 03.07.2026: настоящий диагональный рез у уклона (не только учёт в раскрое)

Продолжение предыдущего фикса. Раньше кусок, задевающий уклон, кроился
как прямоугольник по худшей (более высокой) из двух точек колонки —
раскрой был "верный по счёту материала в целом", но конкретно этот
кусок завышал расход и не показывал, где реально проходит рез.

**Новый общий модуль `src/core/geometry2d.ts`** — универсальные 2D
примитивы, НЕ привязанные к ГКЛ и вообще к материалу: `polygonArea()`
(площадь многоугольника, формула шнурков) и `clipRectBySlopedTop()`
(обрезка прямоугольника наклонной линией, Sutherland-Hodgman по одной
полуплоскости). Осознанно вынесено из `profileGeometry.ts` (тот —
про геометрию именно профиля стены) в отдельный файл общего назначения,
т.к. следующий материал с раскроем (когда дойдёт очередь) сможет
переиспользовать эти же функции без изменений.

**В `calcSheetLayout.ts`:** кусок, который реально пересекает наклонную
линию колонки (не просто "колонка наклонная", а конкретно этот кусок
задет), получает `kind: 'diagonal_cut'` и настоящий `polygon`
(треугольник/трапеция/пятиугольник) вместо прямоугольника. Площадь
такого куска в `usedAreaM2`/`wastePercent` считается по многоугольнику,
а не по `w*h` — то есть реальные потери материала у самого уклона
теперь видны в проценте отходов, а не скрыты внутри "верного по счёту"
округления.

**В `SheetLayoutCanvas.tsx`:** такой кусок рисуется настоящим
многоугольником (Konva `Line` с `closed`), а исходная заготовка
(прямоугольник, из которого он вырезан) показана пунктиром вокруг —
видно и что реально пойдёт на стену, и сколько места кусок занял на
листе/в пуле обрезков.

**Важно на будущее (сказано пользователем прямо):** этот же класс
геометрических задач планируется и для листовых материалов, которые
появятся позже (пока в раскрое только ГКЛ/ГВЛ/Сапфир/Аквамарин — они
уже все проходят через один и тот же `calcSheetLayout`, значит фикс
уже работает для всех четырёх). Также заявлена будущая потребность
в дугах/окружностях ("нужен циркуль" — скорее всего скруглённые углы
или круглые проёмы на холсте). **Это ДРУГАЯ геометрическая область** —
пересечение отрезка с линией (сделано) и пересечение с окружностью
(не сделано) считаются разными функциями. Если возьмёмся за дуги —
им место в том же `geometry2d.ts` (уже есть комментарий-заглушка
в шапке файла), но код придётся писать с нуля, ничего не переиспользуется
напрямую из уклона.

**Диагональный физический рез листа по месту всё ещё делает монтажник
руками** — программа не советует "куда конкретно приложить рулетку",
она верно считает материал и рисует, где проходит линия. Этого
достаточно для сметы и раскроя; более детальная разметка под рез —
не обсуждалось, не в скоупе.

## Багфикс 03.07.2026: раскрой листов ГКЛ игнорировал уклон

**Симптом:** если в калькуляторе (App.tsx/CeilingCalc или LiningCalc.tsx)
задавался уклон потолка/пола (ломаная EdgeProfile), сам расчёт материала
(площадь ГКЛ, длины направляющих) уклон учитывал корректно — а вот
раскрой листов (`calcSheetLayout` → визуальная раскладка листов на
стене, «сколько листов и где резать») уклон игнорировал полностью.

**Причина:** `calcSheetLayout` принимал одну плоскую `wallH: number` —
вызывающий код (`App.tsx`, `LiningCalc.tsx`) заранее сворачивал профиль
в `maxStudHeight()` (одну "худшую" точку) и отдавал только её. Раскрой
кроил ВСЮ стену по этой одной высоте, как будто уклона нет вообще.

**Фикс:** `calcSheetLayout` теперь принимает `ceilingProfile`/`floorProfile`
(EdgeProfile) вместо плоской `wallH`. Точки перегиба обоих профилей
добавлены в границы колонок раскроя (`columnBoundaries`, как и края
проёмов) — внутри каждой колонки высота стены теперь линейна. Высота
колонки = худшая из высот на её двух краях (не худшая по всей стене).
Диагональный рез у самого уклона остаётся ручной подгонкой монтажника —
это осознанно не решается (см. геометрию ниже), но количество листов
и расположение горизонтальных стыков теперь верно следует форме уклона.

⚠️ **API изменился** — если где-то ещё встретится вызов
`calcSheetLayout(wallL, wallH, ...)` со старой сигнатурой (плоское
число вторым аргументом) — это будет ошибка типов, нужно завернуть
плоскую высоту в `flatProfile(wallL, h)` + `flatProfile(wallL, 0)`.
Так уже сделано в `calcProjectSheetLayout.ts` (проектный раскрой с
плана — там профили пока всегда плоские, отдельный незакрытый пункт
дорожной карты, см. ниже про `planLineToSurfaceInput`).

Добавлены тесты `calcSheetLayout.test.ts` (7 шт, раньше у файла не
было тестов вообще) — включая регрессионный тест именно на уклон.

## ДОЛГОСРОЧНОЕ ВИДЕНИЕ ПРОЕКТА (зафиксировано 03.07.2026, НЕ в работе)

Пользователь (монтажник/прораб, не программист — держать объяснения
простыми в диалоге с ним) сформулировал развёрнутую конечную цель.
Сознательно отложено ради текущих задач (раскрой/материалы/баги),
но должно быть на виду для планирования порядка следующих слоёв.

**Идея одной фразой:** RTS-стратегия, перенесённая на реальную стройку.
Холст — не чертёж, а живая модель объекта. Каждый объект на плане
(стена, перегородка, плитка, потолок) — не линия, а сущность со
своими свойствами и статусом. Ресурсы (материалы) считаются сразу,
как только объект появился на холсте (это уже частично работает —
расчётный движок готов, просто не для всех типов объектов и не
привязан к статусам).

**Статусы объекта** (сейчас `getStatus()` — заглушка, всегда 'none'):
- существующий (стоит и было до нас)
- неактивный (запланирован, работа не началась)
- в работе
- закончен
- заблокирован — с причиной (нет материала / ждём смежников / другое).
  Причина блока — отдельное поле, не просто ярлык, пригодится для
  списка "что сейчас держит стройку".

**3D-вид, реагирующий на статус в реальном времени.** Меняешь статус
линии на "в работе" — на вкладке 3D конструкция начинает возводиться
прямо на глазах. "Существующие" конструкции в 3D уже стоят изначально.
Технически: 3D-сцена строится из тех же данных, что и холст (материал,
высота, проёмы — всё уже есть в моделе `PlanLine`), просто нужен
рендер-слой поверх них. **В проекте пока нет НИ ОДНОЙ 3D-библиотеки** —
это будет отдельный крупный кусок работы, не мелкая фича.

**Совместная работа в реальном времени.** "Все, кто участвует в работе
над проектом" видят изменения статусов друг друга сразу. База есть
(Supabase, уже используется для авторизации/хранения проектов), но
сам realtime-канал (подписка на изменения) пока не подключён.

**Шаблоны однотипных квартир.** Если фирма делает ремонт в одинаковых
квартирах — не перерисовывать план заново на каждую, а иметь шаблон,
"проштамповать" его N раз с нумерацией, и в 3D эти копии встают друг
на друга как этажи. Логически независимо от статусов/3D-движка,
может делаться параллельно или отдельной веткой позже.

**Согласованный порядок (по итогам разговора):** статусы — это
фундамент, без него не имеет смысла начинать 3D (нечего будет туда
передавать кроме голой геометрии). 3D-движок — самый затратный по
времени слой, начинать точно не сейчас. Шаблоны квартир — не
зависят от первых двух, можно рассмотреть отдельно, если появится
практическая нужда раньше.

**Что спросить в начале сессии, если разговор зайдёт про это снова:**
готовы ли наконец взяться за статусы (это открывает путь к 3D), или
всё ещё приоритет — текущий расчётный функционал.

## Идея 03.07.2026: свой визуальный стиль отрисовки профиля (НЕ в работе)

Разговор начался с вопроса пользователя про фото реального стыка ПС75×50×0.6
(митровый угол на объекте, зенитный фонарь) — не баг, а вопрос "насколько
сложно сделать отрисовку стоек ближе к реальной геометрии, но со своим
узнаваемым стилем, и в 2D, и в будущем 3D". Зафиксирован план на будущее,
без даты начала:

1. **Точное сечение профиля вместо плоского прямоугольника** (низкая
   сложность, дни, не недели). Сейчас стойка на канвасе — один прямоугольник
   с линейным градиентом (имитация объёма). Реальный профиль — гнутый лист:
   спинка + 2 полки (отгиб) + 2 губки (короткий возвратный загиб на концах).
   Рисуется параметрически один раз (по размерам из `profiles.ts`), дальше
   переиспользуется. Штамповку-насечку (ромбик на плоскостях жёсткости) —
   через SVG/Konva `Pattern`, не руками на каждую стойку.
2. **Свой фирменный стиль** (та же сложность, что и пункт 1, другое
   решение). Не фотореализм, а плоская заливка + акцентная полоса-маркер
   (цвет полосы = тип профиля ПС50/75/100, толщина полосы = толщина
   металла 0.6/0.7) + тики-засечки как на линейке. Более читаемо при
   мелком зуме на телефоне на объекте, чем реализм.
3. **Настоящий 3D** — другая лига трудозатрат, отдельный крупный кусок
   работы (в проекте пока нет ни одной 3D-библиотеки, см. "ДОЛГОСРОЧНОЕ
   ВИДЕНИЕ" выше). Предложенный путь: смоделировать эталонный отрезок
   профиля один раз в Blender (бесплатный, ближе к CAD-точности, чем к
   рисованию — extrude 2D-сечения вдоль пути + материал "оцинковка"),
   экспорт в `.glb`, переиспользовать как ассет везде, где нужен 3D-вид.
   Начинать не раньше, чем появится сама 3D-сцена (статусы → 3D, см. выше).

**Рекомендованный порядок, если возьмёмся:** сначала пункт 1 (не блокирует
остальную разработку, мгновенный визуальный эффект), пункт 2 отдельным
разговором про палитру/акценты, пункт 3 — только когда дойдёт очередь
до самой 3D-сцены по долгосрочному видению, не раньше (иначе ассет
делается впустую, негде показать).

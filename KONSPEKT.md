Фикс (коммит c3c940f):

Итог: дерево материалов реально у левого края окна, канвас растягивается

## Что сделано в предыдущем чате (хронологически)

### 1. Убраны все визуальные маркеры на концах линий
Круги на концах (select-handles, hover-точки, snap-точки) визуально ломали
восприятие плана при мелком масштабе — выглядело как "круги с чёрточкой",
а не как конструкция. Убрано полностью:
- Drag-handles остались, но стали прозрачной хитзоной (radius=12, invisible)
- Snap cursor: круг → крестик, масштабируется 7/stageScale (не растёт с зумом)
- ChainStartPt: большой круг → маленькая точка 4/stageScale

### 2. Wall join — геометрия стыков двойных линий
Новый модуль `src/core/wallJoin.ts`. Три типа стыков:
- **L (угол, обе линии заканчиваются в одной точке)**: митровый стык через
  пересечение параллельных граней (Side+/Side−)
- **T (одна линия упирается в тело другой)**: примыкающая обрезается до
  **ближней** грани главной (не оси! — важный фикс, см. ниже)
- **Продолжение (коллинеарные)**: шов не рисуется

Приоритет в L-угле (когда обе линии кончаются в одной точке, нет естественной
"главной"): длиннее — сквозная; при равной длине — нарисованная раньше.
Правило подтверждено пользователем на реальном примере монтажа перегородок.

Рендер двойных линий переписан: вместо одного прямоугольника — заливка по
"расширенной" оси (без дыр в углах) + граничные линии по join-точкам +
торцы только на реально свободных концах.

### 3. Подложка PDF — обводка чужого проекта (калька)
- `src/core/pdfBackground.ts`: рендер страницы PDF → PNG dataURL через pdf.js
  (`renderScale=2`, worker подключен через Vite `?url`-импорт)
- `types.BackgroundImage` в `FloorPlan`: dataUrl, x/y/w/h, opacity, locked
- Стор: `setBackgroundImage`/`updateBackgroundImage`, персистится в localStorage
- Левая панель: загрузка PDF (кнопка), выбор страницы если многостраничный,
  слайдер прозрачности, кнопка калибровки, удаление подложки
- Калибровка переиспользует уже существовавший режим `scale` (клик-клик-ввод мм)
- **isPointAllowed**: ограничение "рисовать только внутри замкнутого периметра"
  ПОЛНОСТЬЮ снимается если `floorPlan.backgroundImage` задан — при кальке
  геометрия уже задана исходным чертежом, не нужно сначала замыкать Room
- Подтверждено в этом чате: подложка двигается/скроллится зажатой средней
  кнопкой мыши через общий механизм панорамирования канваса (тот же код,
  что и Space+ЛКМ) — отдельная логика перемещения самой подложки не нужна,
  пункт "подложка не двигается" в багах снят как неактуальный

### 4. Глобальная перекалибровка масштаба по факту
Идея пользователя: рисуем весь план по приблизительному масштабу, затем
выбираем ЛЮБУЮ уже начерченную линию, вводим её точный реальный размер —
масштаб всего плана пересчитывается, обновляются lengthMm/areaM2/perimeterMm
у ВСЕХ линий/помещений/контуров. Координаты x1,y1,x2,y2 не трогаются.
`recalibrateByLine(lineId, exactMm)` в FloorPlan.tsx. UI: поле "точный
размер, мм" + кнопка "Уточнить масштаб" в шапке инспектора любой линии.

### 5. Серия фиксов T-снапа (самая долгая часть чата)
Пользователь: "перегородка не прилипает к ребру другой стены, зазор ~180мм".
Прошли несколько итераций, каждая вскрывала следующий слой проблемы:

1. **snapPoint видел только концы линий**, не тело — добавлена проекция
   точки на отрезок (T-примыкание к середине стены)
2. **Порог снапа не учитывал толщину стены** — на толстых перегородках
   (ГКЛ ps100) клик по видимому краю выходил за фиксированный экранный
   порог. Добавлен `halfThicknessPx` к порогу.
3. **Реальный баг сравнения**: `best.d` стартовал с `threshPx` и "сжимался"
   из-за ДРУГИХ, не относящихся линий — валидный кандидат с толщинным
   порогом отбрасывался. Переписано: `best.d = Infinity`, каждый кандидат
   валидируется относительно СВОЕГО порога.
4. **Клик перехватывался обработчиком той самой стены** — у каждой линии
   есть `onMouseDown` для select/drag, который безусловно делал
   `cancelBubble=true` даже в режиме 'draw'. Клик просто не доходил до
   `handleStageClick`. Фикс: в режимах draw/scale обработчик линии ничего
   не делает.
5. **Концептуальная ошибка подхода**: снап целился в ОСЬ главной стены,
   а wall-join потом пытался "дорисовать" примыкание до грани — это давало
   и неверную длину (до центра, не до грани — "в свету"), и в самой
   дорисовке была перепутана сторона (extSign). **Исправлено кардинально**:
   T-снап теперь целится СРАЗУ в БЛИЖНЮЮ ГРАНЬ стены (не ось), длина
   получается верной без доп. коррекции. wallJoin.applyT extSign тоже
   поправлен для старых (уже нарисованных до фикса) линий.

Итог: T-снап работает надёжно, длина "в свету" считается автоматически.
**Важно для будущих сессий**: если пользователь опять пожалуется на снап —
проверять именно эту цепочку (видит ли snapPoint тело линии → учитывает
ли толщину → не сжался ли best.d раньше времени → не перехватывает ли
handleLinePointerDown клик → целится ли в грань, а не в ось).

### 6. Расширение спецификации ГКЛ: тип листа + слои
Существующий справочник `GklSubtype` (стандарт/влагостойкий ГКЛВ/
огнестойкий ГКЛО/влагоогнестойкий ГКЛВО) переиспользован как третье
измерение спецификации. `PlanLineSpec` расширен: `boardSubtype?`, `layers?`.
Облицовки переосмыслены: раньше "1/2 слоя" было единственным подтипом,
теперь подтип — способ монтажа (на клею/каркас ПН28/ПС50/ПС75), а слои —
отдельное независимое поле. UI: каскадные селекторы в левой панели
(после выбора профиля/монтажа) и в инспекторе (ConstructionSpecSelector).

### 7. Дверные и оконные проёмы
`PlanOpening { id, type: door|window, offsetMm, widthMm, heightMm,
sillHeightMm? (только окно — низ от УЧП), label }`. `PlanLine.openings?`.

`computeOpeningSegments()` в FloorPlan.tsx разбивает ось стены на сплошные
сегменты вокруг проёмов — рендер двойной линии переписан с одного
прямоугольника на цикл по сегментам, торцы автоматически появляются на
краях проёма. Подпись (Д-1/О-1, сквозная нумерация) рисуется в разрыве.

Инспектор: блок "Проёмы" — список с удалением + форма добавления
(тип дверь/окно кнопками, отступ+ширина+высота числами, для окна
доп. поле "низ от пола"). Дефолты: дверь 2000мм высота, окно 1200×900.
Валидация: offset+width не должны превышать длину линии.

### 8. Высота конструкции — от хардкода к гибкости
Было: `calcLineArea` использовал константу 3000мм везде, известный баг
из конспекта "высота в таблице — хардкод 3000мм".

Стало в два шага:
- `PlanLine.heightMm?` — редактируемое поле, доступно в таблице
  "Конструкции на плане" (инлайн input) и в инспекторе
- `calcLineArea` использует `l.heightMm ?? 3000`, площадь проёма
  считается по ЕГО СОБСТВЕННОЙ высоте (widthMm×heightMm), не высоте стены

Затем пользователь уточнил: 95%+ перегородок на объекте одной высоты
(конкретный пример 3620мм), редактировать каждую линию постфактум
неудобно. Добавлено: `drawHeightMm` — глобальное поле "Высота, мм" в
левой панели НАД деревом материалов, дефолт 3000, держится между
рисованием линий. Оба места создания линии (обычный клик, замыкание
цепочки) передают это значение. Для исключений — правка в инспекторе
как раньше, ничего не убрано.

## Известные особенности / договорённости для будущих сессий

1. **При наличии подложки PDF ограничение "сначала периметр" снято** —
   можно рисовать любую конструкцию в любом порядке. Это специально для
   сценария кальки, не для рисования с нуля.

2. **T-снап целится в грань стены, не в ось.** Длина примыкающей стены
   автоматически "в свету". Если пользователь рисовал ДО фикса (коммит
   17b5044) — его старые линии останутся с завышенной длиной (до центра),
   пересчитать можно через recalibrateByLine.

3. **Offset проёма отсчитывается от ПЕРВОЙ точки линии** (x1,y1) —
   зависит от того, в какую сторону пользователь чертил стену.

4. **drawHeightMm — глобальный, не привязан к материалу.** Один рубильник
   на весь сеанс рисования, не сбрасывается при смене материала/типа.

5. Подложка PDF рендерится один раз при загрузке (`renderScale=2`,
   фиксированный размер) — не пересчитывается, не двигается отдельно
   от плана, но панорамируется вместе с планом (см. п.3 в разделе
   "Новое в этом чате" — подтверждено, работает через общий пан канваса).

6. Wall-join расширяет ось линии под стыки (`ax1/ay1/ax2/ay2` в
   `JoinedWall`) — это ВЛИЯЕТ на `computeOpeningSegments`, которая
   аккуратно берёт исходные (не расширенные) `l.x1/l.y1` для отсчёта
   offsetMm проёма, но расширенные `fax1..fay2` для крайних сегментов
   заливки. Если будете трогать один модуль — проверяйте другой.

7. **maxWidth: 900 у обёртки контента остаётся для вкладок
   Перегородки/Облицовка/Потолки** (формы) — сознательно не тронут,
   раньше уже обсуждали, что для узких полей это ок. Убран только для
   вкладки «План» (см. раздел "Новое в этом чате").

## Архитектура данных (types/index.ts) — актуальный срез

```typescript
interface PlanLineSpec {
  material: string
  subtype?: string
  boardSubtype?: GklSubtype   // стандарт/влагостойкий/огнестойкий/влагоогнестойкий — для gkl
  layers?: 1 | 2               // кол-во слоёв обшивки — для gkl
}

interface PlanOpening {
  id: string
  type: 'door' | 'window'
  offsetMm: number    // от x1,y1 линии
  widthMm: number
  heightMm: number     // дверь — от пола; окно — высота самого окна
  sillHeightMm?: number // только окно — низ от УЧП
  label: string
}

interface PlanLine {
  id: string
  x1: number; y1: number; x2: number; y2: number
  type: PlanLineType
  lengthMm: number
  label: string
  spec?: PlanLineSpec
  wallId?: string; liningId?: string
  openings?: PlanOpening[]
  heightMm?: number    // дефолт 3000 если не задано
}

interface BackgroundImage {
  dataUrl: string       // PNG base64
  x: number; y: number
  width: number; height: number
  opacity: number
  locked: boolean
}

interface FloorPlan {
  scaleMmPerPx: number
  lines: PlanLine[]
  contours: PlanContour[]
  rooms: Room[]
  backgroundImage?: BackgroundImage | null
}
```
